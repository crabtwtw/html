SECTION 04

申請審核（同意 / 拒絕）

客戶 /start 後送 UID + API Key，會出現 admin bot 通知。你只要按 ✅ / ❌ 按鈕即可。

v24 改動 ❌ 拒絕 3 個 reject 觸發點都統一改 DELETE
v23 vs v24 行為對照
項目v23（舊）v24（新）
DB 變化 status='rejected'
API Key 殘留		 整筆 DELETE
API Key 完全清除
客戶端訊息 ❌ 申請未通過，
請聯繫管理員		 ❌ 申請未通過。
如需重新申請，請輸入 /start
客戶感受 以為被永久封鎖 知道可以重來
3 個 Reject 觸發點
# 不管在哪個階段被拒絕，都會 DELETE 整筆
reject_uidonly  # 客戶送 UID 後拒絕
reject_uid      # 完整 UID 申請拒絕
reject_api      # 客戶送 API Key 後拒絕（最敏感）
注意

拒絕 = 整筆 DELETE，不可逆。誤按拒絕沒關係，客戶 /start 重新申請即可。

SECTION 10

常見情境流程

🌅 情境 A · 早上開機處理新申請

1. /users pending ← 看有幾個待審核 2. 對每個待審核： ✅ 同意 → 進下一階段（系統自動處理） ❌ 拒絕 → 整筆 DELETE（v24） 3. /users active ← 確認新增了幾個 active

🔍 情境 B · 客戶私訊問問題

1. 客戶給你 BingX UID 12345678 2. /uid 12345678 ← 反查到 chat_id 3. /user <chat_id> ← 看詳情 4. /pnl <chat_id> ← 看他的損益狀況 5. /orders <chat_id> ← 看他目前的持倉

🚨 情境 C · 客戶說 API Key 失效要重設

1. /user <chat_id> 2. 按【🔑 刪除 API(重設)】按鈕 3. 系統自動引導客戶重新輸入 API Key 4. 客戶送新 API Key 後你再 ✅ 同意

📢 情境 D · 系統要維護

1. /maint ← 切換維護模式 或 /bc 系統 23:00-00:00 維護，請暫停下單 2. 維護完成後再 /maint 切回

🔎 情境 E · 找特定客戶（500+ 用戶時）

1. /users search 王 ← 用名字找 或 /users search 78626 ← 用部分 ID 找 2. 從列表找到目標 chat_id 3. /user <chat_id> ← 進入管理

💰 情境 F · 月底看整體績效

1. /stats ← 整體本月損益 2. /users active ← 確認 active 客戶數 3. （手動 export 客戶見證、規劃下月行銷）
SECTION 11

七大協作準則

這套系統能做到 v24 級別，背後是 七條 Mountain 準則。未來再改時請對照這七條。

01
寧可賺少不要爆倉
風險哲學的核心。帳戶不能歸零，單筆少賺沒關係，永遠不能讓客戶死掉。
02
改動給 diff 確認才推
透明原則。所有改動先給 diff 看，等同意才動，不能默默改了就推上去。
03
V1 V2 永遠一起動
一致性原則。V1 和 V2 訊號處理是對稱的雙胞胎，改一個一定要改另一個。
04
數字用 python 算不要猜
實證原則。任何數字、計算、比例都要用 python 跑過，不能憑感覺講「大概是 X」。
05
最小改動，不自作聰明加功能
節制原則。只解該解的問題，不主動加「客戶可能會喜歡」的東西。
06
重大改動前先分析風險
謹慎原則。動手前列出可能副作用、給多個方案 A/B/C 比較、等選哪個方案才動。
07
猜是 Mountain 的權利，AI 必須驗證
AI 紀律。不能猜（任何不確定都要 python 模擬驗證）；不能擅自做（有想法先講，等同意）；有意見要說（不裝沒意見）；發現錯要認（不裝沒看到）。違反就違反準則。
核心精神

「Mountain 訂方向，AI 跑驗證；改之前先看 diff，動之前先分析；V1 V2 一起動，最小改動不亂加；數字用 python 算，不確定不能猜；客戶安全永遠第一。」
